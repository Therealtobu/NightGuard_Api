Protection Layers

Every layer makes
reversal harder.

NightGuard V2 stacks 10 independent protection passes, each targeting a different analysis vector.

Custom VM Bytecode

Source compiles to a proprietary instruction set. The original Lua is never present in the output — only encrypted bytecode executed by an embedded VM.

Opcode Aliasing

Every logical opcode has 1–3 aliases that share the same value. Opcode numbers shuffle per build, completely breaking pattern-matching deobfuscators.

Rolling-XOR Strings

String literals are encrypted with a rolling-XOR key schedule: each byte's cipher key depends on the previous encrypted byte. Decrypted only inside the VM.

Control Flow Flattening

if/else blocks are converted to state-machine dispatch loops. The logical program flow is completely destroyed from a static-analysis perspective.

Opaque Predicates + Dead Code

Conditions that always evaluate to true/false but look dynamic. Dead code blocks inject noise that real analysis tools must reason through.

Dual-Layer Bytecode Encryption

Serialized bytecode passes through rolling-XOR cipher then a 32-byte key-block XOR. Both layers must be reversed in the correct order before execution.

No Loadstring

Unlike most obfuscators, NightGuard never calls loadstring. The VM interprets bytecode directly — no string eval attack surface.

Per-Build Randomization

Every single build has different opcode values, different encryption keys, and different junk distributions. The same source produces unique output each time.

Semantically Identical

The protected script runs exactly like the original. NightGuard's VM faithfully implements all Lua 5.1 semantics including closures, varargs, and metatables.

FAQ

Common questions.

How is this different from other Lua obfuscators?
Most obfuscators wrap code in loadstring() with encoded strings. NightGuard V2 compiles to a custom bytecode format and executes it through an embedded VM — there is no Lua source in the output at all, only encrypted binary instructions.
Will the script still run the same way?
Yes. The NightGuard VM faithfully executes all Lua 5.1 semantics. Functions, closures, tables, varargs, string operations, and global access all work identically to the original.
Is the web tool running Python?
The web tool runs a JavaScript preview that applies variable renaming, string encoding, and dead code injection in the browser. For full VM bytecode compilation, download the Python CLI and run: python cli.py input.lua output.lua
Does it support Roblox Luau?
Yes. The VM runtime is written in Lua 5.1 compatible code with a compatibility shim for bitwise XOR that works across Lua 5.1, LuaJIT, and Roblox Luau environments.
Can I obfuscate multiple files?
Each file is obfuscated independently. For multi-module projects, run cli.py on each module separately, then assemble in Roblox Studio as usual.
